# 天气识别比赛：Agent 工作可用信息提取

> 目标：把赛事规则与 FAQ 中对“自动化参赛 Agent / 辅助开发 Agent / 训练推理流水线 Agent”有用的信息提取出来，便于后续做任务拆解、代码生成、训练调参、提交检查和风险控制。

\---

## 1\. 比赛任务定义

### 1.1 赛题名称

* 智海算法调优
* CAIP 强脑赛道 / 智能算法赛项 / 智海算法调优赛题

### 1.2 核心任务

实现一个 **天气图片分类模型**，对输入图片进行分类。

### 1.3 分类类别

官方明确为 **四分类**：

|类别|
|-|
|多云|
|雨天|
|雪天|
|晴天|

注意事项：

* FAQ 中明确不是六类，是四类。
* MO 平台上如果看到六类测试集，FAQ 说明那不是官方数据。
* 类别顺序需要以官方 baseline / 官方数据说明为准，不能自行假定。

### 1.4 图片类型

* 图片类似手机拍摄的普通天气图片。
* 不是卫星云图。
* 图片来自互联网，包含多种拍摄视角和场景。
* 图片尺寸没有统一标准，需要进行数据预处理。

## 2\.  需要完成的核心工作

从自动化角度，可以把 Agent 工作拆成以下模块。

### 2.1 数据理解

负责：

* 读取训练集目录结构或标注文件。
* 识别类别映射。
* 统计每类样本数量。
* 检查是否类别不平衡。
* 检查图片尺寸、格式、损坏图片。
* 生成数据集分析报告。

输出建议：

```text
dataset\\\_report.md
class\\\_distribution.csv
bad\\\_images.txt
label\\\_mapping.json
```

### 2.2 数据预处理 

负责：

* 统一图片读取逻辑。
* 设计训练集和验证集 transforms。
* 处理不同尺寸图片。
* 处理灰度图、RGBA 图、损坏图等异常情况。
* 确保训练和推理阶段的预处理一致。

推荐预处理方向：

```python
Resize
CenterCrop / RandomResizedCrop
ToTensor
Normalize
```

训练增强可以考虑：

```python
RandomResizedCrop
RandomHorizontalFlip
ColorJitter
RandomRotation
RandAugment
MixUp
CutMix
```

注意：

* 天气分类依赖颜色、光照、天空纹理，增强不能过度。
* ColorJitter 不能太激进，否则会破坏天气语义。
* 雪天、雨天、多云、晴天之间可能存在较强视觉混淆，需要增强策略谨慎设计。

### 2.3 模型选择

负责：

* 选择适合 CPU 推理的图像分类模型。
* 对比准确率、macro F1、推理速度、模型大小。
* 生成模型候选表。
* 自动切换 backbone 做实验。

推荐优先尝试：

|模型|特点|
|-|-|
|ResNet18 / ResNet34|稳定、训练简单、CPU 推理较快|
|EfficientNet-B0 / B1|精度和速度较均衡|
|MobileNetV3|轻量，适合 CPU 推理|
|ConvNeXt-Tiny|精度可能较好，但 CPU 速度需要测试|
|Swin-Tiny|可尝试，但 CPU 推理压力可能更大|

不建议一开始使用过大的模型，因为最终评测是 CPU 推理，并且有总时间限制。

### 2.4 训练调参 

负责：

* 自动训练不同模型。
* 自动记录超参数。
* 自动保存最优权重。
* 自动计算 macro F1。
* 生成混淆矩阵。
* 针对弱类别做调参。

核心指标：

```text
macro F1
per-class precision
per-class recall
per-class F1
confusion matrix
```

重点原因：

* 比赛评分使用 F1 分数。
* FAQ 明确是 macro F1。
* macro F1 会平均关注每个类别，少数类表现差会明显拉低最终分。

建议关注的混淆关系：

|易混淆类别|原因|
|-|-|
|多云 vs 晴天|光照、天空颜色可能接近|
|雨天 vs 多云|阴天、路面不明显时容易混|
|雪天 vs 晴天|高亮雪景可能被误判|
|雨天 vs 雪天|灰暗场景、低对比度图像可能相似|

### 2.5 推理性能 

负责：

* 在 CPU 上测试推理速度。
* 统计整体推理耗时。
* 检查是否能在 70 分钟内完成评分集推理。
* 记录模型大小、内存占用、CPU 占用。
* 优化 batch size、num\_workers、图像读取方式。

约束：

```text
最终推理使用 CPU
测试总时长应在 70 分钟内
评分集大概几千张图片
推理时间是整体时间，不是单张图片时间
```

应该估算：

```text
总推理时间 = 平均单张推理时间 × 评分集图片数量 + 数据读取与预处理开销
```

### 2.6 提交检查

负责最终提交前的自动检查。

必须检查：

* 推理代码能否独立运行。
* 权重文件路径是否正确。
* 是否不依赖本地绝对路径。
* 是否能在 CPU 环境运行。
* 依赖库是否可安装。
* 输出格式是否符合平台要求。
* 类别映射是否正确。
* 推理时间是否小于 70 分钟。
* 是否没有调用外部 API。
* 是否提交了推理代码和模型权重。

建议输出：

```text
submit\\\_checklist.md
submission\\\_smoke\\\_test.log
cpu\\\_inference\\\_benchmark.csv
```

## 5\. 数据与提交规则

### 5.1 数据集开放时间

* 预选赛素材 / 数据集开赛时提供。
* 赛前只能在 MO 平台课程板块下实训项目进行了解。

### 5.2 数据划分

FAQ 中给出比赛数据逻辑：

|数据集|说明|
|-|-|
|训练集|选手可获得，用于训练|
|测试集|用于选手代码测试和模型效果评估|
|评分集|隐藏数据，用于最终推理评分|

### 5.3 是否需要标注

* 不需要自己标注数据。

### 5.4 评分集规模

* FAQ 说明评分数据集大概有几千张。

### 5.5 提交内容

最终提交：

```text
推理代码 + 模型权重
```

不是只提交 F1 值，也不是只提交代码。

### 5.6 提交次数

* 比赛测试次数无限制。
* 正式提交只有一次。

Agent 需要重点防止：

* 提交前没有完整 smoke test。
* 权重路径写错。
* 类别顺序错。
* 预处理和训练不一致。
* CPU 推理超时。
* 误把调试代码提交上去。

## 6\. 评分规则与排序规则

### 6.1 省赛评分

规则文件给出：

```text
最终得分 = F1 分数 × 100
```

### 6.2 F1 类型

FAQ 明确：

```text
macro F1
```

这意味着每个类别的 F1 会被平均计算，不能只优化总体 accuracy。

### 6.3 同分排序

如果最终得分相同：

1. 按模型推理时间由短到长排序。
2. 如果推理时间仍相同，优先考虑代码执行效率，例如内存占用率、CPU/GPU 利用率。
3. 如果仍无法区分，由组委会审核代码规范性，例如注释完整性、代码结构清晰度。

Agent 因此需要额外关注：

* 推理速度。
* 内存占用。
* 代码结构。
* 注释清晰度。
* 可复现性。
* 提交包简洁度。

## 7\. 允许与禁止事项

### 7.1 允许

|行为|说明|
|-|-|
|使用开源模型|FAQ 明确允许|
|使用预训练模型权重|FAQ 明确允许|
|安装其他库|FAQ 明确允许|
|本地训练后上传|FAQ 明确允许|
|使用大模型辅助编程|FAQ 明确允许|
|多次测试|测试次数无限制|

### 7.2 禁止 / 高风险

|行为|风险|
|-|-|
|外接大模型 API 做推理|FAQ 明确不可以|
|调用外部 API 完成分类|不符合“仅限制在深度学习模型”的要求|
|攻击系统或窃取数据|直接违规|
|抄袭或影响公平性|可能取消资格|
|异常提交|组委会保留审查权|

Agent 的合规边界：

* 可以辅助写代码、分析数据、调参。
* 不能在最终推理时调用外部大模型 API。
* 不能访问未授权数据。
* 不能自动攻击、爬取、绕过平台限制。
* 不能把隐藏评分集作为训练数据使用。

## 8\. 对设计最重要的约束

### 8.1 最终推理必须 CPU 可运行

因此 不能只根据 GPU 验证集 F1 选模型，还要加入 CPU benchmark。

建议每个实验记录：

```yaml
model\\\_name:
input\\\_size:
params:
weight\\\_size\\\_mb:
val\\\_macro\\\_f1:
val\\\_accuracy:
cpu\\\_batch\\\_size:
cpu\\\_total\\\_time\\\_estimate:
avg\\\_latency\\\_ms:
memory\\\_mb:
```

### 8.2 正式提交只有一次

Agent 应设计为“保守提交”：

1. 先做本地 / 平台 smoke test。
2. 再跑小规模推理。
3. 再跑全量测试集推理。
4. 再检查输出格式。
5. 再压缩提交包。
6. 最后人工确认提交。

### 8.3 macro F1 优先级高于 accuracy

Agent 选择模型时应使用：

```text
primary metric: macro F1
secondary metric: CPU inference time
third metric: model size / memory
```

### 8.4 类别映射是高风险点

Agent 必须从官方数据或 baseline 中自动读取 label mapping。

禁止硬编码未经确认的类别顺序。

### 8.5 图片尺寸不统一

需要写鲁棒 dataset loader：

* 支持 jpg / jpeg / png / bmp 等常见格式。
* 支持 RGB 转换。
* 异常图片跳过或记录。
* transform 在 train / val / test / submit 阶段一致可控。
